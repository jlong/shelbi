```text
shelbi agent <SUBCOMMAND> [OPTIONS]
```

`shelbi agent` manages the [agents](/docs/concepts/agents) a project ships
with — the roles, each a system prompt plus an optional skill set, that a
[workspace](/docs/concepts/workspaces) loads when it picks up a task. Every
project starts with four (`orchestrator`, `developer`, `qa`,
`security-review`); these subcommands surface them and let you author more.

Agents live under `~/.shelbi/projects/<project>/agents/<name>/`, each with an
`instructions.md` (its prompt) and an optional `skills/` directory. The
shared `agents/_shared/preamble.md` is prepended to every agent's prompt. The
defaults are seeded on first load and never clobbered on upgrade — see
[customizing an agent](/docs/concepts/agents#customizing-an-agent).

Every subcommand accepts the global `-p / --project <PROJECT>` flag. It's
omitted from the per-subcommand tables below to keep them readable.

## list

```text
shelbi agent list
```

Print every agent in the project, one per line, with whether it's a shipped
default or one you authored, and which workflow statuses reference it via
their [`agent:` field](/docs/concepts/workflows#owners-and-agents).

```text
NAME              SOURCE     USED BY
orchestrator      default    (window 1)
developer         default    Todo, InProgress
qa                default    Review
security-review   default    —
```

An agent with no referencing status (`—`) is defined but unused — fine for
one you're about to wire into a workflow.

## show

```text
shelbi agent show <NAME>
```

Print an agent's **resolved** prompt to stdout — `_shared/preamble.md`
prepended to the agent's own `instructions.md`, with `{{assistant_name}}`
substituted, exactly as the runner would receive it. This is the quickest way
to see what an agent will actually be told, preamble included. Pass
`--raw` to print just the agent's `instructions.md` without the preamble.

| Flag | Type | Default | Description |
| --- | --- | --- | --- |
| `<NAME>` | string | — | Agent name (positional, required). |
| `--raw` | flag | off | Print only `instructions.md`, skipping the shared preamble and substitution. |

## new

```text
shelbi agent new [OPTIONS] <NAME>
```

Scaffold a new agent directory under `agents/<NAME>/` with a starter
`instructions.md` and an empty `skills/`. The starter prompt is a minimal
role template, not a copy of `developer` — it's there to be replaced.

Errors if an agent with that name already exists. Names must be non-empty,
must not start with `.`, and may only contain `a-z`, `0-9`, `-`, `_` — the
name doubles as the identifier referenced from workflow `agent:` fields, so
the validator is stricter than POSIX.

| Flag | Type | Default | Description |
| --- | --- | --- | --- |
| `<NAME>` | string | — | Agent name (positional, required). |
| `--edit` | flag | off | Open the new `instructions.md` in `$EDITOR` after creating it. |

## edit

```text
shelbi agent edit <NAME>
```

Open an agent's `instructions.md` in `$EDITOR` (defaults to `vi`). If the
named agent is a shipped default that hasn't been materialized to disk yet,
its default `instructions.md` is written out first so you have something
concrete to tweak. Any other missing name errors with a hint to run
`shelbi agent new <NAME>` first.

Edits take effect the next time the affected agent is launched — for the
orchestrator, on the next `shelbi reload`; for a workspace agent, on its next
task dispatch.

## Examples

See the project's agents and where each is used:

```bash
shelbi agent list
```

Read exactly what the QA agent will be told, preamble and all:

```bash
shelbi agent show qa
```

Bake your repo's conventions into the default developer agent:

```bash
shelbi agent edit developer
```

Scaffold a focused reviewer and open it straight away:

```bash
shelbi agent new perf-review --edit
```

Then wire it into a workflow status:

```yaml
statuses:
  - { name: Perf, category: handoff, owner: agent, agent: perf-review }
```
