Shelbi doesn't impose a branching model. A **[Shelbi Workflow](/docs/concepts/workflows)**
— the per-project YAML in `workflows/<name>.yaml` — is a declarative
description of the statuses a task moves through and the git side-effects
that fire on each transition. Point it at whatever branching model your
team already runs and Shelbi automates that model instead of a new one.

These guides each take one well-known git branching model and show the
Shelbi Workflow that implements it: the statuses and their categories, the
transitions that push, PR, and merge branches in the right place, and any
adjustment the [orchestrator prompt](/docs/getting-started/custom-workflow)
needs to route and merge the way the model expects.

## What a Shelbi Workflow encodes

Three layers do the work, and every guide fills in the same three:

- **Statuses** — the columns on the board. Each declares a
  [category](/docs/concepts/workflows#status-categories) (`backlog`,
  `ready`, `active`, `handoff`, `done`) that generic code reasons in, an
  `owner` (`user` or `agent`), and an optional `agent:` role.
- **Transitions** — the git side-effects (`push_branch`, `open_pr`,
  `merge`, `close_pr`, `delete_branch`, `restack`) that fire when a task
  crosses an edge. This is where a branching model actually lives: *what*
  lands *where*.
- **Orchestrator prompt** — the [routing and merge judgment](/docs/getting-started/custom-workflow)
  that isn't structural enough to be YAML. Which branch a task targets,
  when auto-merge is allowed, how contributions get triaged.

## The models

| Model | Shape | Guide |
| ----- | ----- | ----- |
| **Feature-branch** (GitHub-flow) | Branch per task off `main`, PR, merge. The default Shelbi shape. | [Feature-branch →](/docs/guides/feature-branch) |
| **Trunk-based** | Short-lived branches, merge to `main` fast. Pairs naturally with Zen auto-merge. | [Trunk-based →](/docs/guides/trunk-based) |
| **Git-flow** | Long-lived `develop` integration branch with `feature`/`release`/`hotfix` branches. | [Git-flow →](/docs/guides/git-flow) |
| **Forking** | Contributions arrive from forks; a review gate stands between them and `main`. | [Forking →](/docs/guides/forking) |
