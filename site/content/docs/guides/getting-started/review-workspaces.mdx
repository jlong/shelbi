The [multi-workspace loop](/docs/guides/getting-started/multi-workspace) gets
finished branches back to you — but reading a diff isn't the same as *running*
the change. Most real review of an app or a site means booting the dev server,
opening a URL, and clicking through it. A **review workspace** is the pool slot
that does exactly that: it loads a finished task's branch, installs and builds
it, and serves it on a port so you can inspect the running app.

This page walks the whole flow end to end — declare the slot, reload, get a
task to the review status, load it, open the served URL, and accept or bounce.
For the model underneath it (ports, the `review:` config block, the Review
agent, the load → serve lifecycle), see the reference:
[Review workspaces](/docs/concepts/review-workspaces).

<Callout type="note" title="You need a working pool first">

This page assumes you already have a project with a workspace pool and can
promote a task through the board — the [first task](/docs/guides/getting-started/first-task)
and [multi-workspace](/docs/guides/getting-started/multi-workspace) pages get
you there. A review workspace is an *addition* to that pool, not a replacement.

</Callout>

## Set up and use the review flow

<Steps>
<Step title="Declare a review workspace">

A review workspace is an ordinary `workspaces:` entry with one extra key:
`role: review`. Open your project's YAML — `~/.shelbi/projects/<name>.yaml` in
[global mode](/docs/concepts/config-modes), `<repo>/.shelbi/project.yaml`
in-repo — and add one to the pool:

```yaml
workspaces:
  - { name: alpha,    machine: hub, runner: claude }
  - { name: bravo,    machine: hub, runner: claude }
  - { name: review-1, machine: hub, runner: claude, role: review }
```

`role` takes `dev` (the default, elided from the wire form) or `review`. A slot
with no `role:` is a normal development workspace, so every existing project
YAML keeps working untouched. Because `role` rides the **user-local** config
half alongside the rest of `workspaces:`, each person decides how many review
slots their own machines carry.

<Callout type="warning" title="Two review workspaces per machine, max">

Review workspaces are **scarce by design: 1–2 per machine**, because each one
may hold a running server bound to a port. That's a hard invariant — the config
loader refuses to start a project that declares a third review slot on any one
machine. One is plenty to start with. See [the scarcity
invariant](/docs/concepts/review-workspaces#the-scarcity-invariant) for why.

</Callout>

Most projects need nothing else: the Review agent auto-detects how to install,
build, and serve from your project shape (`package.json`, `Cargo.toml`, a
`Makefile`, a `Procfile`). If yours needs explicit commands or a non-default
port, add a [`review:` block](/docs/concepts/review-workspaces#the-review-config-block)
— but leave it out until you find you need it.

</Step>
<Step title="Reload to pick it up">

```bash
shelbi reload
```

The new slot appears under **Workspaces** in the sidebar, grouped under its
machine and reading `idle` until a task loads onto it.

</Step>
<Step title="Get a task to the review status">

Run a task the way you normally would — promote a card and let a dev workspace
pick it up, or dispatch one through the orchestrator. When the dev workspace
finishes it writes its review-ready marker; the hub poller rebases the branch
onto the base branch and moves the task into the **Review** status. (That
`handoff`-category status is a [workflow](/docs/guides/getting-started/workflows)
concept — the board column a task sits in awaiting human input.)

The finishing dev workspace then closes its own session and returns to `idle` —
it doesn't linger. Its work is now waiting to be *run*, not edited.

</Step>
<Step title="Load it onto a review workspace">

If your project declares any review workspace, the **orchestrator auto-loads**
the task onto a free review slot the moment it reaches the review status — the
same path you can trigger by hand:

```bash
shelbi review dark-mode-toggle-polish
```

Either way, the Review agent takes over on the review slot: it installs deps,
builds, starts the dev server on the slot's assigned port, waits for the ready
probe, and posts a "ready" note with the URL. When a slot is free you'll see:

```text
→ loading dark-mode-toggle-polish onto review workspace review-1 on hub (:3000)
✓ review agent up at shelbi-myapp:review-1
```

If every review slot on the machine is already busy, nothing fails and nothing
is preempted — the task waits in Review, marked *pending-load*, and the command
reports its queue position:

```text
⏳ all review workspaces on hub are busy — `dark-mode-toggle-polish` is queued at position 1
   review-1 → palette-fuzzy-match-fix
It will load automatically when a review workspace frees.
```

<Callout type="note" title="Declare a slot before you review">

Review workspaces are the only review surface. `shelbi review <id>` on a
project that declares none gives a loud onboarding error telling you to add a
`role: review` slot and reload — it never silently falls back. See
[`shelbi review`](/docs/cli/review) for the full command reference.

</Callout>

</Step>
<Step title="Open the served URL and click through">

Select the task under **Ready for Review** in the sidebar (or press `Enter` on
its review-workspace row) to focus that workspace's window — the managed server
pane sits beside the agent pane. Open the served URL,
`http://localhost:3000` here, and click through the running change the way a
user would.

The Review agent's charter is narrow: **load and serve, don't code.** It won't
modify the branch or move the card — the one exception is if you explicitly ask
for a tweak ("bump the header padding"), which it makes in place and refreshes.

</Step>
<Step title="Accept or bounce">

You own the decision. Moving the task out of Review is the signal:

- **Accept** — move the card to **Done** (`L` in the kanban view, or
  `shelbi task move dark-mode-toggle-polish --to done`). Per your
  [workflow](/docs/guides/getting-started/workflows), that transition merges the
  branch back into the default branch.
- **Bounce** — send it back to **To Do** (`H`, or
  `shelbi task move dark-mode-toggle-polish --to todo`) for another pass.

Either move tears down the server pane and returns the review workspace to
`idle`, which immediately loads the next task queued for review, if any.

</Step>
</Steps>

## Ready vs Queued for Review

The sidebar splits review-status tasks into two sections, so a glance tells you
what's running from what's waiting:

```text
 — Ready for Review —
 ✓ Polish dark-mode toggle       hub:3000
   shelbi/dark-mode-toggle-polish

 — Queued for Review —
 · Palette fuzzy-match fix
   shelbi/palette-fuzzy-match-fix
```

- **Ready for Review** (`✓`, cyan) — loaded on a review workspace and serving.
  Line 1 carries the `machine:port` URL; line 2 the branch. Selecting one
  focuses that review workspace's window so you can inspect the running app.
- **Queued for Review** (`·`, dim) — a review-status task waiting for a free
  review slot. No URL yet because nothing is serving it; it loads automatically
  the moment a slot frees.

That split is the whole payoff of the scarcity invariant: with only 1–2 review
slots per machine, the queue is where you see review pressure build.

## Next

Routing a task onto a review workspace is the *human* path — the whole point is
human eyes on the running app — so a review-loaded task is deliberately **not**
eligible for Zen Mode's auto-merge. When you're ready to hand the orchestrator
the cases it *can* judge on its own,
[enable Zen Mode](/docs/guides/getting-started/enable-zen-mode) to auto-promote
backlog and auto-merge finished branches that clear a confidence bar you define.
