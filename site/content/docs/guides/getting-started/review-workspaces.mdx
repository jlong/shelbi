The [multi-workspace loop](/docs/guides/getting-started/multi-workspace) gets
finished branches back to you. But reading a diff isn't the same as *running*
the change. Most real review of an app or a site means booting the dev server,
opening a URL, and clicking through it.

Shelbi has no special "review mode" for this — you assemble it from three
generic primitives: a **workspace carries a tag**, a **status requires that
tag**, and a **transition runs commands**. This page wires them together end
to end. For the model underneath, see the reference:
[Review workspaces](/docs/concepts/review-workspaces).

<Callout type="note" title="You need a working pool first">

This page assumes you already have a project with a workspace pool and can
promote a task through the board. The [first task](/docs/guides/getting-started/first-task)
and [multi-workspace](/docs/guides/getting-started/multi-workspace) pages get
you there. A review workspace is an ordinary slot with a tag, not a new kind of
thing.

</Callout>

## Set up and use the review flow

<Steps>
<Step title="Tag a workspace for review">

Add a `review` tag to a slot in your project YAML
(`~/.shelbi/projects/<name>.yaml` in [global mode](/docs/concepts/config-modes),
`<repo>/.shelbi/project.yaml` in-repo). Tag one workspace directly, or tag the
machine so every slot on it inherits the tag:

```yaml
workspaces:
  - { name: alpha,    machine: hub, runner: claude }
  - { name: bravo,    machine: hub, runner: claude }
  - { name: review-0, machine: hub, runner: claude, tags: [review], slot: 0 }
```

`tags` accepts a bare string as shorthand (`tags: review`). The `slot: 0` makes
the numeric slot explicit so the transition commands below can derive a stable
port from `$SLOT`; leave it out and it defaults to the slot's index among the
machine's workspaces.

<Callout type="tip" title="One review slot is plenty to start">

Each review slot may hold a running server bound to a port, so give the port a
per-slot offset (`$((3000 + $SLOT))`) if you add a second. One is enough for a
first pass.

</Callout>

</Step>
<Step title="Require the tag on your review status and boot a server">

In your [workflow file](/docs/configuration/workflow), require the `review` tag
on the review status so tasks route to the tagged slot, and add transitions
that boot the dev server on entry and tear it down on exit:

```yaml
statuses:
  - { id: in-progress, owner: agent, agent: developer }
  - { id: review,      owner: user,  agent: review, tags: [review] }
  - { id: done,        owner: user }

transitions:
  - from: in-progress
    to: review
    run:
      - npm install
      - npm run dev -- --port $((3000 + $SLOT)) &
    ready: curl -fsS http://localhost:$((3000 + $SLOT)) > /dev/null
    ready_timeout: 120
  - from: review
    to: done
    actions: [merge, delete_branch]
    run:
      - pkill -f "port $((3000 + $SLOT))" || true
  - from: review
    to: in-progress
    run:
      - pkill -f "port $((3000 + $SLOT))" || true
```

The enter transition installs deps, launches the server (backgrounded, so the
command returns), and `ready` polls the URL until it answers. Both exit edges
tear the server back down. Swap the `npm` lines for whatever your project
needs — `cargo run`, a `Makefile` target, a `Procfile web:` line.

</Step>
<Step title="Reload to pick it up">

```bash
shelbi reload
```

The tagged slot appears under **Workspaces** in the sidebar, grouped under its
machine and reading `idle` until a task routes onto it.

</Step>
<Step title="Get a task to the review status">

Run a task the way you normally would: promote a card and let a dev workspace
pick it up, or dispatch one through the orchestrator. When the dev workspace
finishes it writes its review-ready marker; the hub poller rebases the branch
onto the base branch and moves the task into the **Review** status.

Because that status requires `tags: [review]`, the orchestrator routes the task
onto a free `review`-tagged slot (preferring the one it ran on) and fires the
enter transition: install, boot the server on the slot's port, wait for
`ready`. If every matching slot is busy the task waits in Review until one
frees — nothing is preempted.

The finishing dev workspace closes its own session and returns to `idle`. Its
work is now waiting to be *run*, not edited.

</Step>
<Step title="Open the served URL and click through">

Select the task under **Review** in the sidebar (or press `Enter` on its
workspace row) to focus that workspace's window. Open the served URL —
`http://localhost:3000` for `slot: 0` here — and click through the running
change the way a user would.

If you named the built-in `review` agent on the status, its charter is narrow:
**load and serve, don't code.** It won't modify the branch or move the card.
Point `agent:` at an agent you've authored to change what runs there.

</Step>
<Step title="Accept or bounce">

You own the decision. Moving the task out of Review is the signal, and each
move fires that edge's transitions — including the teardown `run`:

- **Accept** — move the card to **Done** (`L` in the kanban view, or
  `shelbi task move dark-mode-toggle-polish --to done`). Per the transitions
  above, that edge merges the branch, deletes it, and tears the server down.
- **Bounce** — send it back to **In Progress** (or **To Do**) for another pass;
  that edge tears the server down too.

Either move returns the workspace to `idle`, which loads the next task routed
for review, if any.

</Step>
</Steps>

## Next

Routing a task to a review workspace is the *human* path: the whole point is
human eyes on the running app. Keeping `merge` on the *accept* edge and off the
*enter*-review edge means the orchestrator holds a review task for you rather
than auto-merging it. When you're ready to hand the orchestrator the cases it
*can* judge on its own,
[enable Zen Mode](/docs/guides/getting-started/enable-zen-mode) to auto-promote
backlog and auto-merge finished branches that clear a confidence bar you define.
