An **agent** is a *role*: a system prompt plus a set of skills. It is the
"who" of a piece of work — the orchestrator that schedules, the developer
that writes the code, the reviewer that checks it. An agent is deliberately
*not* the task it's handed, and *not* the slot it runs in. Those are separate
things:

- The **task** is the work — a markdown card with a prompt and a branch.
- The **[workspace](/docs/concepts/workspaces)** is the capacity — a tmux
  pane plus a git worktree on some machine.
- The **agent** is the role — the prompt and skills that decide *how* the
  work gets done.

Splitting these apart is what lets the same workspace run a developer agent
on one task and a reviewer agent on the next, and what lets a
[workflow](/docs/concepts/workflows#owners-and-agents) say "this status is reviewed by
the security agent" as a line of YAML rather than a paragraph of prompt prose.

## The three shipped agents

Every project ships with three agents — `orchestrator`, `developer`, and
`review`. They bracket the loop end to end: schedule the work, build it, serve
the result for a human. They're the defaults the generic machinery wires up
unless you say otherwise.

| Agent | Role |
| --- | --- |
| `orchestrator` | The agent you talk to. Turns requests into cards, dispatches tasks to free workspaces, tails the events log, reports back. Runs in window 1, never in a workspace. See [Orchestrator](/docs/concepts/orchestrator). |
| `developer` | The default workspace agent. Implements a task on its branch, runs the project's checks, writes the review-ready marker. This is the agent an `agent`-owned status uses when no other is named. |
| `review` | The agent that runs on a [review workspace](/docs/concepts/review-workspaces). It loads a finished branch and *serves* it — installs deps, builds, boots the dev server — then hands a human a working URL. It doesn't write code (bar an explicitly requested tweak). |

`developer` runs most often; `orchestrator` and `review` bracket it.

Reviewer roles aren't shipped, but they're a `shelbi agent new` away — and
because an agent is just a role, an authored one slots into the loop exactly
like a built-in. Two common ones:

- **`qa`** — exercises a finished branch against acceptance criteria before it
  reaches a human. Point a `handoff`-category status at it.
- **`security-review`** — a focused adversarial pass for sensitive diffs (auth,
  migrations, install scripts). Pair it with a status, or a danger-path-heavy
  workflow, where the extra latency is worth it.

## Where agents live on disk

An agent is a directory under the project's Shelbi config:

```text
~/.shelbi/projects/<project>/agents/
├── _shared/
│   └── preamble.md          # project-wide context, prepended to every agent
├── orchestrator/
│   └── instructions.md      # the orchestrator's system prompt
├── developer/
│   ├── instructions.md      # the developer agent's system prompt
│   └── skills/              # agent-scoped skills (optional)
├── review/
│   ├── instructions.md      # the Review agent's system prompt
│   └── skills/              # ships a `load-run-detection` skill
└── qa/                      # ← one you authored with `shelbi agent new`
    └── instructions.md
```

Each agent has at minimum an `instructions.md` — its system prompt. It may
also carry a `skills/` directory of agent-scoped skills the runner loads when
that agent is active. The `_shared/preamble.md` file is special: its contents
are prepended to *every* agent's instructions, so project-wide context (the
repo layout, the house style, the test command) lives in one place instead of
being copy-pasted into four prompts.

When a task is dispatched, the agent named for that status (or `developer` by
default) has its rendered prompt — `_shared/preamble.md` followed by the
agent's own `instructions.md` — handed to the runner in the workspace. The
[orchestrator's own prompt](/docs/concepts/orchestrator#how-the-prompt-is-wired)
is rendered the same way; it's just the agent that happens to run in window 1.

## Customizing an agent

Agents are **project-local with shipped defaults**. The first time a project
loads, Shelbi materializes the three default agents into the directory above.
From then on they're yours to edit — change `developer/instructions.md` to
bake in your repo's conventions, drop a skill into a role's `skills/`, tighten
an authored `security-review` prompt to your threat model.

The defaults are *seeded, not enforced*. On upgrade, Shelbi only writes an
agent file that doesn't already exist — it never clobbers a file you've
edited. New shipped agents (or new skills) show up; your customizations stay
put. If you want to reset an agent to the current default, delete its
`instructions.md` and reload.

Edit an agent by hand, or through the CLI:

```bash
shelbi agent list                 # the three shipped, plus any you've added
shelbi agent show developer       # print instructions.md + skills list
shelbi agent edit developer       # open instructions.md in $EDITOR
shelbi agent new qa               # author a reviewer role
```

See the [`shelbi agent`](/docs/cli/agent) reference for the full command set.

## Assigning an agent to work

You rarely assign an agent to a task directly. Instead, a
[workflow](/docs/concepts/workflows#owners-and-agents) status declares who owns it:

```yaml
# workflow file — reference-only; name/category live in statuses.yml
statuses:
  - { id: todo,        owner: agent, agent: orchestrator }
  - { id: in-progress, owner: agent, agent: developer }
  - { id: review,      owner: agent, agent: qa }
  - { id: done,        owner: user }
```

`owner: agent` makes a status the orchestrator's to act on; the optional
`agent:` field names *which* agent does it. Omit `agent:` and the
`developer` agent is used. The `review` status above is auto-reviewed by a
`qa` agent you've authored before a human ever sees it — that's the whole "a
reviewer is a role you can drop on a status" idea, expressed declaratively. See
[owners](/docs/concepts/workflows#owners-and-agents) for the rules.
