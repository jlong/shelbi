## Where Shelbi excels

### Persistent workers, not one-shot runs

Shelbi keeps each worker pinned to its own git worktree across tasks. The
worker's tools, indexes, and shell state survive between cards. Example: a
worker that's already cloned the repo and warmed its build cache picks up
the next ready task without re-paying setup cost.

### You own the orchestration loop

The orchestrator runs as a Claude agent on your machine, tailing
`~/.shelbi/events.log` and dispatching work. Swap models, edit the
project's `CLAUDE.md`, or replace the orchestrator entirely — there is no
hosted control plane in the loop.

### Review-first, not merge-first

Finished work lands in a `review` column with the branch ready to inspect.
The default is *don't merge until a human signs off*. Use the review pane
in the TUI to read the diff before promoting to `done`.

## Choose Shelbi if…

- You want a Kanban-shaped queue across multiple machines you already own.
- You're comfortable in tmux and want workers that survive SSH drops.
- Reviewing diffs before merge is non-negotiable.
- You want the orchestration logic to be inspectable code in your repo.

## Choose Example Tool if…

- You'd rather pay for hosted infrastructure than run workers yourself.
- Your team already lives inside Example Tool's surface and switching costs are high.
- A specific Example Tool feature is load-bearing for your workflow.
