# Release Distribution: Homebrew + APT

## Context

Shelbi currently installs from source with `./scripts/install.sh`: it runs `cargo build --release`, copies `target/release/shelbi` to `$HOME/bin/shelbi`, and re-signs the copied binary ad-hoc on macOS so the kernel will execute it. That is acceptable for contributors, but it is not a release mechanism. Users need stable package-manager installs, reproducible artifacts, and a way to verify what they installed.

This plan designs the first public binary distribution path:

- macOS via a dedicated Homebrew tap.

- Ubuntu via a signed APT repository.

- GitHub Releases as the canonical immutable artifact ledger for each version.

The recommendation is to use GoReleaser for release orchestration and Rust builds, nFPM through GoReleaser for `.deb` packages, and a custom repository-publishing step for APT metadata. GoReleaser has first-class Rust builder support, can publish Homebrew casks/formula-like tap files after a GitHub release, can produce checksums, can sign archives/packages/checksums, and can drive nFPM package generation. APT repository metadata still needs explicit generation and signing with `apt-ftparchive` or `reprepro`.

- GoReleaser Rust builder: <https://goreleaser.com/customization/builds/builders/rust/>

- GoReleaser Homebrew publishing: <https://goreleaser.com/customization/publish/homebrew_casks/>

- GoReleaser nFPM packaging: <https://goreleaser.com/customization/package/nfpm/>

- GoReleaser checksums and signing: <https://goreleaser.com/customization/package/checksum/> and <https://goreleaser.com/customization/sign/sign/>

- Homebrew Formula Cookbook: <https://docs.brew.sh/Formula-Cookbook>

- Debian `apt-secure(8)`: <https://manpages.debian.org/bookworm/apt/apt-secure.8.en.html>

- Debian `sources.list(5)`: <https://manpages.debian.org/bookworm/apt/sources.list.5.en.html>

## Decision Summary

1. Canonical version source is the workspace package version in `crates/shelbi-cli/Cargo.toml`, because that crate owns the `shelbi` binary. Release tags must be `vMAJOR.MINOR.PATCH` and must exactly match that Cargo version.
2. Canonical release source of truth is the signed Git tag plus the GitHub Release for that tag. Package managers only point to artifacts published there or to repo metadata generated from those artifacts.
3. Initial macOS distribution uses a dedicated tap, `jlong/homebrew-shelbi` or `shelbi/homebrew-shelbi` depending on org ownership. Do not start in `homebrew-core`; core adds review overhead and prefers mature formulae with stable demand.
4. Initial Homebrew package should be a formula that installs prebuilt tarballs for macOS arm64 and macOS x86\_64. Use a cask only if Shelbi later ships a `.pkg`, `.app`, launchd service installer, or notarized installer bundle.
5. Initial Ubuntu distribution uses `.deb` packages generated by nFPM and an APT repository published from GitHub Pages or an object-storage bucket under a stable HTTPS domain.
6. Sign release checksums and APT repository metadata with separate keys. Prefer keyless Sigstore for release-artifact attestations when available, but do not make APT depend on Sigstore because APT needs OpenPGP-signed repository metadata.
7. Rollback means republishing package-manager metadata to point at the previous good version. Do not mutate or delete GitHub Release artifacts except for a documented security incident.

## Release Source of Truth

### Versioning

Use semantic versions without build metadata for public releases:

- Tag format: `v0.4.0`, `v1.2.3`.

- Cargo version source: `crates/shelbi-cli/Cargo.toml` package `version`.

- The release workflow fails if `GITHUB_REF_NAME` without the leading `v` differs from that Cargo version.

- Release candidates, if needed, use tags like `v0.4.0-rc.1`; these publish GitHub prereleases but do not update Homebrew stable or APT stable metadata.

Add a small validation script in `scripts/release/check-version.sh`:

```bash
# !/usr/bin/env bash
set -euo pipefail
version=$(cargo metadata --no-deps --format-version 1 | jq -r '.packages[] | select(.name == "shelbi-cli") | .version')
tag=${GITHUB_REF_NAME:-}
tag_version=${tag#v}
[[ "$tag" == v* ]] || { echo "release tag must start with v" >&2; exit 1; }
[[ "$version" == "$tag_version" ]] || { echo "Cargo version $version != tag $tag_version" >&2; exit 1; }
```

### GitHub Release Artifacts

For each non-prerelease tag, publish at least:

- `shelbi_Darwin_arm64.tar.gz`

- `shelbi_Darwin_x86_64.tar.gz`, if CI can build and smoke-test x86\_64 or if cross-compilation is validated

- `shelbi_Linux_x86_64.tar.gz`

- `shelbi_0.4.0_amd64.deb`

- `checksums.txt`

- `checksums.txt.sig` or `checksums.txt.pem`/Sigstore bundle, depending on signing choice

- `attestation.intoto.jsonl` or GitHub artifact attestations, if enabled

- SBOM files if added later; not required for the first release but should be in the workflow shape

Each tarball contains:

```text
shelbi
LICENSE
README.md
```

Do not put generated shell-completion files in the first release unless Shelbi already has a stable completion generator. Add them as a later package enhancement.

### Signing

Use three trust layers:

1. Git tag signing: maintainers sign release tags with SSH or GPG. CI verifies the tag maps to the checked-out commit.
2. Artifact integrity: publish SHA256 checksums and sign the checksum file. Prefer Sigstore keyless signing in GitHub Actions for low secret burden; use a GPG signing key only if maintainers need offline-verifiable signatures immediately.
3. APT trust: use a dedicated OpenPGP APT repository signing key. Store the private key as a GitHub Actions secret and publish the dearmored public key at `https://apt.shelbi.dev/shelbi-archive-keyring.gpg` or equivalent.

Keep the APT signing key separate from developer tag-signing keys. Rotate APT keys by publishing a `shelbi-archive-keyring` package later; for the first release, publish the key file and fingerprint in docs.

## Automation Choice

### Recommended: GoReleaser + nFPM + Custom APT Publish

Use GoReleaser because it fits the current repo shape:

- The project is a Rust workspace with a single user-facing binary.

- GoReleaser now has a Rust builder path, so the release configuration can describe target triples and artifacts instead of hand-maintaining multiple shell scripts.

- It can produce archives, checksums, signed artifacts, GitHub Releases, and Homebrew tap updates in one release transaction.

- Its nFPM integration covers Debian package layout and metadata without introducing Debian packaging boilerplate on day one.

Keep APT repository publication custom because nFPM creates packages, not a complete signed APT repository. The repository step is small and explicit:

1. Place `.deb` files under `pool/main/s/shelbi/`.
2. Generate `dists/stable/main/binary-amd64/Packages` and `Packages.gz`.
3. Generate `dists/stable/Release`.
4. Sign `Release` to produce both `InRelease` and `Release.gpg`.
5. Publish to GitHub Pages or object storage.

### Alternative: Custom GitHub Actions Scripts

Custom scripts would avoid a GoReleaser dependency, but the repo would own more packaging edge cases: cross-target naming, archive layout, checksum generation, signing, tap PRs, nFPM invocation, and release retries. That is not worth it for the first implementation.

### Alternative: cargo-dist

`cargo-dist` is attractive for Rust-only release artifacts and installers, but this plan needs Homebrew tap updates, Debian package metadata, and a signed APT repository. It may be a future replacement if it cleanly owns those outputs, but it is not the lowest-risk first path.

## Homebrew Design

### Tap Strategy

Start with a dedicated tap:

```bash
brew tap shelbi/shelbi
brew install shelbi
```

Use `homebrew-shelbi` as the GitHub repository name. Homebrew maps `shelbi/homebrew-shelbi` to `shelbi/shelbi` for users.

Do not submit to `homebrew-core` initially. Revisit core after several stable releases, issue volume is manageable, the release process is boring, and the project has enough external demand to justify core review.

### Formula Shape

Use a formula rather than a cask for the first release because Shelbi is a CLI binary, not an app bundle or graphical installer.

Target shape:

```ruby
class Shelbi < Formula
  desc "Terminal-native orchestrator for teams of coding agents"
  homepage "https://github.com/jlong/shelbi"
  version "0.4.0"
  license "MIT"

  on_macos do
    if Hardware::CPU.arm?
      url "https://github.com/jlong/shelbi/releases/download/v0.4.0/shelbi_Darwin_arm64.tar.gz"
      sha256 "..."
    else
      url "https://github.com/jlong/shelbi/releases/download/v0.4.0/shelbi_Darwin_x86_64.tar.gz"
      sha256 "..."
    end
  end

  def install
    bin.install "shelbi"
  end

  test do
    assert_match version.to_s, shell_output("#{bin}/shelbi --version")
  end
end
```

Keep Linux Homebrew out of scope for the first tap unless users ask for it; Ubuntu users should use APT.

### Update Automation

The release workflow updates the tap after GitHub Release artifacts exist and checksums are finalized.

Preferred implementation:

- Use GoReleaser Homebrew publishing if it can generate a formula with the exact prebuilt-binary shape needed.

- If GoReleaser's built-in Homebrew publisher only fits casks cleanly for the selected version, use a small `scripts/release/update-homebrew-formula.rb` script and push a commit to the tap repo.

Required tap automation behavior:

- It updates only `Formula/shelbi.rb`.

- It commits `shelbi 0.4.0` with the release actor identity.

- It opens a PR for review at first; switch to direct push only after two clean releases.

- It runs `brew audit --strict --online shelbi` and `brew test shelbi` in the tap workflow.

## APT Design

### Hosting Strategy

Use a static APT repository under a stable HTTPS URL. Preferred first host:

- GitHub Pages from a dedicated repository, e.g. `shelbi/apt`, served as `https://apt.shelbi.dev/`.

Why GitHub Pages first:

- It is simple, static, cheap, and easy to inspect.

- The repository metadata is small.

- The signing model does not rely on GitHub for trust; APT validates signed metadata.

Move to object storage plus CDN later if package download volume or custom-domain needs outgrow GitHub Pages.

### Repository Layout

Use a standard two-level archive layout:

```text
apt-repo/
  shelbi-archive-keyring.gpg
  pool/main/s/shelbi/shelbi_0.4.0_amd64.deb
  dists/stable/InRelease
  dists/stable/Release
  dists/stable/Release.gpg
  dists/stable/main/binary-amd64/Packages
  dists/stable/main/binary-amd64/Packages.gz
```

Start with:

- Suite: `stable`

- Component: `main`

- Architecture: `amd64`

- Package name: `shelbi`

Add `arm64` only after Linux arm64 is built and smoke-tested in CI or on a reliable runner.

### `.deb` Metadata

Generate `.deb` packages with nFPM. Package metadata:

- Package: `shelbi`

- Version: release version without leading `v`

- Section: `devel`

- Priority: `optional`

- Architecture: `amd64` initially

- Maintainer: maintainer or project email; open decision

- Homepage: public Shelbi docs URL or GitHub repo

- License: MIT

- Depends: `tmux (>= 3.2), git, openssh-client`

- Recommends: none initially

- Suggests: `codex, claude` only if package names exist in target repositories; otherwise document agent CLIs outside package metadata

- Installed files: `/usr/bin/shelbi`, `/usr/share/doc/shelbi/README.md`, `/usr/share/licenses/shelbi/LICENSE` or Debian-appropriate copyright file

Do not install or enable the Shelbi daemon from package maintainer scripts in the first version. Package installation should put the binary on disk only. Users can run `shelbi daemon install` explicitly. Auto-enabling user services from a system package is surprising and has edge cases across desktop/headless systems.

### User Install Commands

Docs should publish commands like:

```bash
sudo install -d -m 0755 /etc/apt/keyrings
curl -fsSL https://apt.shelbi.dev/shelbi-archive-keyring.gpg \
  | sudo tee /etc/apt/keyrings/shelbi-archive-keyring.gpg >/dev/null

echo "deb [arch=amd64 signed-by=/etc/apt/keyrings/shelbi-archive-keyring.gpg] https://apt.shelbi.dev stable main" \
  | sudo tee /etc/apt/sources.list.d/shelbi.list >/dev/null

sudo apt update
sudo apt install shelbi
```

Also document manual verification:

```bash
curl -fsSLO https://github.com/jlong/shelbi/releases/download/v0.4.0/checksums.txt
curl -fsSLO https://github.com/jlong/shelbi/releases/download/v0.4.0/shelbi_0.4.0_amd64.deb
sha256sum -c checksums.txt --ignore-missing
```

For APT metadata verification, users rely on `apt update`; APT refuses unsigned repositories by default and verifies signed Release metadata.

### APT Signing

Generate metadata with `apt-ftparchive` or `reprepro`. For the first implementation, prefer `apt-ftparchive` because the repository is tiny and the generated files are easy to review.

Signing commands in CI:

```bash
gpg --batch --yes --clearsign -o dists/stable/InRelease dists/stable/Release
gpg --batch --yes -abs -o dists/stable/Release.gpg dists/stable/Release
```

Required Release fields:

- `Origin: Shelbi`

- `Label: Shelbi`

- `Suite: stable`

- `Codename: stable`

- `Architectures: amd64`

- `Components: main`

- `Description: Shelbi APT repository`

## CI / Release Workflow

### Workflows

Add `.github/workflows/release.yml` triggered by:

```yaml
on:
  push:
    tags:
      - 'v*.*.*'
```

Job outline:

1. `validate` on Ubuntu:

   - Checkout full history.

   - Install Rust stable.

   - Run `scripts/release/check-version.sh`.

   - Run `cargo test --workspace`.

   - Run site/package checks only if release docs or site files changed; otherwise release should not depend on Node.
2. `build` matrix:

   - `macos-14` for `aarch64-apple-darwin`.

   - `macos-13` or equivalent Intel runner for `x86_64-apple-darwin`, if available. If not available, explicitly defer x86\_64 macOS support instead of pretending it is tested.

   - `ubuntu-22.04` or `ubuntu-24.04` for `x86_64-unknown-linux-gnu`.

   - Each job builds release binary, runs `shelbi --version`, and uploads artifacts.
3. `release`:

   - Runs GoReleaser with `--clean` after all builds pass, or lets GoReleaser own the build matrix if its Rust builder is configured for all targets.

   - Publishes GitHub Release artifacts, checksums, signatures, and `.deb` package.
4. `homebrew`:

   - Updates the dedicated tap after GitHub Release upload succeeds.

   - Uses a fine-scoped token that can write only the tap repository if possible.
5. `apt`:

   - Downloads the released `.deb` artifact.

   - Checks checksum against `checksums.txt`.

   - Updates the static APT repository.

   - Signs `Release` into `InRelease` and `Release.gpg`.

   - Publishes to the `apt` repository's Pages branch or bucket.
6. `verify-install`:

   - On macOS: `brew tap ...`, `brew install shelbi`, `shelbi --version`.

   - On Ubuntu container/VM: add keyring and source list, `apt update`, `apt install shelbi`, `shelbi --version`.

### Required Secrets

- `GITHUB_TOKEN`: default token for release creation in the main repo, if permissions allow.

- `TAP_GITHUB_TOKEN`: token or GitHub App credentials that can write to `homebrew-shelbi`.

- `APT_REPO_TOKEN`: token or GitHub App credentials that can write to the APT hosting repo, if separate from the main repo.

- `APT_GPG_PRIVATE_KEY`: ASCII-armored private key for signing APT metadata.

- `APT_GPG_PASSPHRASE`: passphrase for the APT private key.

- `COSIGN_*` or OIDC permissions: only if using Sigstore keyless signing. Prefer OIDC and no long-lived secret.

- Optional future macOS secrets: Apple Developer ID cert, app-specific password, team ID. Not required for the initial formula using tarred CLI binaries, but required if Shelbi later ships notarized `.pkg`/cask artifacts.

Workflow permissions:

```yaml
permissions:
  contents: write
  id-token: write
  attestations: write
```

Restrict write permissions per job where possible.

## Rollback And Recovery

### Bad GitHub Release Artifact

If an artifact is corrupt before announcement, delete the draft release and rerun. After public release, do not mutate assets. Ship a patch version.

### Bad Homebrew Formula

Rollback by committing the previous known-good formula in the tap. Homebrew users can also install a specific version only if versioned formulae are added later; do not add versioned formulae in v1 of this release system.

### Bad APT Package

Rollback by republishing APT metadata without the bad package as the candidate version:

1. Leave the bad `.deb` in `pool/` for auditability unless it is actively harmful.
2. Regenerate `Packages` from the previous good `.deb` set or pin previous good version as highest candidate.
3. Regenerate and sign `Release`, `InRelease`, and `Release.gpg`.
4. Publish a patch release as soon as possible.

APT does not automatically downgrade installed packages. Document manual downgrade once at least two versions exist:

```bash
apt-cache madison shelbi
sudo apt install shelbi=0.4.0
sudo apt-mark hold shelbi
```

### Key Compromise

- Revoke the compromised APT signing key.

- Publish a new key fingerprint through GitHub Releases, website docs, and the tap repo.

- If a keyring package exists by then, ship an update through the old key only if it is still trustworthy; otherwise require manual key replacement.

- Rotate GitHub repository secrets immediately.

## Local Verification For Maintainers

Before tagging:

```bash
cargo test --workspace
cargo build --release --bin shelbi
./target/release/shelbi --version
```

Dry-run release config:

```bash
goreleaser check
goreleaser release --snapshot --clean
```

Verify generated package locally on Ubuntu:

```bash
sudo apt install ./dist/shelbi_0.4.0_amd64.deb
shelbi --version
dpkg -L shelbi
sudo apt remove shelbi
```

Verify Homebrew formula locally:

```bash
brew install --build-from-source --formula Formula/shelbi.rb
brew test shelbi
brew uninstall shelbi
```

Verify APT repository in a container:

```bash
docker run --rm -it ubuntu:24.04 bash
apt update
apt install -y ca-certificates curl gpg
# add Shelbi key/source, apt update, apt install shelbi
```

## Implementation Tasks

1. Release metadata guard

   - Add `scripts/release/check-version.sh`.

   - Add a CI job that compares `v*` tag to `crates/shelbi-cli/Cargo.toml`.

   - Task size: small.

2. GoReleaser baseline

   - Add `.goreleaser.yaml` for the `shelbi` binary.

   - Configure macOS arm64 and Linux x86\_64 first; include macOS x86\_64 only if CI runner support is confirmed.

   - Configure tarball names and `checksums.txt`.

   - Add `goreleaser check` docs or CI validation.

   - Task size: medium.

3. Debian package generation

   - Add nFPM config inside `.goreleaser.yaml`.

   - Package `/usr/bin/shelbi`, README, license/copyright.

   - Add package metadata and dependency list.

   - Smoke-test package install on Ubuntu.

   - Task size: medium.

4. GitHub release workflow

   - Add `.github/workflows/release.yml` for tag-triggered validation, build, release upload, checksum signing, and artifact attestation.

   - Add required repository permissions and environment protection for releases.

   - Task size: medium to large.

5. Homebrew tap bootstrap

   - Create `homebrew-shelbi` repository.

   - Add initial `Formula/shelbi.rb` pointing to a test or first real release.

   - Add tap CI for `brew audit` and `brew test`.

   - Wire release automation to open PRs against the tap.

   - Task size: medium; spans another repo.

6. APT repository bootstrap

   - Create APT hosting repo or bucket.

   - Generate APT signing key and publish public key/fingerprint.

   - Add script for `pool/`, `Packages`, `Packages.gz`, `Release`, `InRelease`, `Release.gpg` generation.

   - Add a verification job that installs Shelbi from the hosted repo in Ubuntu.

   - Task size: large; spans another repo and secrets.

7. Install docs update

   - Update README and site install docs with Homebrew and APT commands.

   - Keep source-build install as contributor path.

   - Include checksum/manual verification notes.

   - Task size: small to medium.

8. Release runbook

   - Add `docs/release.md` or a cstore operational runbook covering tagging, dry runs, rollback, and key rotation.

   - Include exact commands and expected artifacts.

   - Task size: small.

## Risks And Open Decisions

- macOS x86\_64 support depends on available CI runners or validated cross-compilation. Do not publish x86\_64 artifacts unless smoke-tested.

- macOS Gatekeeper behavior for downloaded CLI tarballs should be tested. Current install script re-signs after copy because `cp` invalidates Cargo's ad-hoc signature; release tarballs may need explicit ad-hoc signing before archive creation, and Homebrew may preserve or re-stage binaries differently.

- Maintainer identity and contact email for Debian metadata are not chosen.

- Final GitHub org names are not chosen: `jlong/*` vs `shelbi/*` affects URLs, tap commands, and APT domain.

- APT domain is not chosen. Recommendation is `apt.shelbi.dev`, but any stable HTTPS host works.

- Signing approach for non-APT release artifacts is open: Sigstore keyless is operationally simpler, while GPG signatures are easier for some users to verify offline.

- Package dependency names for agent CLIs should not be encoded until those CLIs have reliable package-manager names.

- Auto-installing the daemon is intentionally deferred; if users expect package install to register services, that needs a separate design because user-level systemd and launchd have different trust and lifecycle behavior.

- Homebrew formula vs cask may need revisiting if notarization or daemon installation becomes part of the macOS package story.
