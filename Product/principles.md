# Principles

How we decide. `vision.md` says where Shelbi is going and `positioning.md` says how we talk about it; this doc is for settling design arguments. Each principle below is derived from decisions already made in `Plans/` and the shipped architecture, not invented after the fact. When a proposal conflicts with one of these, either the proposal changes or the principle does, and changing a principle is a deliberate act, not a side effect of shipping a feature.

Written July 2026.

**Plain files over platforms.** All state is markdown, YAML, and append-only logs on disk. No database, no cloud control plane, no hosted anything. Anything a user can read with cat, they can trust; anything they can edit with vim, they can fix. In practice this means new features must express their state as files first and code second: tasks are markdown with YAML frontmatter, the event stream is one `key=value` line per event, agents are `instructions.md` in a directory. It settled the worker communication design: when the hub daemon is down, workers fall back to appending directly to `events.log` with `O_APPEND` instead of queueing into a broker, because the file is the system and the daemon is just the preferred writer. It also settled in-repo config: a team shares process by committing YAML and markdown to the repo, not by joining a workspace on a server.

**The orchestrator assigns, it doesn't do.** The agent you talk to never has its hands in the code. It writes work up as tasks, picks a workspace and an agent, dispatches, and reacts to events. That is what makes it safe to interrupt and dump on: there is no in-progress edit to derail. In practice, any proposal that has the orchestrator producing artifacts itself is wrong by default; give it a primitive to invoke instead. Zen mode is the sharpest example: even when the orchestrator merges work autonomously, it does so by calling `zen probe`, `zen pr-create`, `zen ci-watch`, and `zen pr-merge`. It judges and sequences; the Rust primitives touch the branch.

**Tasks are the resolution agents work best at.** Every piece of work becomes a scoped task before an agent touches it, and the task file carries everything the work needs: branch, dependencies, params, the body. Agents do their best work on discrete chunks, and task-level resolution lets a big feature and a one-line fix flow through the same system. In practice, features attach to tasks, not to sessions or to agents: dispatch reads as "run task T with agent A in workspace W," events are keyed by task id, and `{{var}}` substitution in workflows resolves from task frontmatter. It settled dependent-branch handling: when a parent task merges, the orchestrator restacks every task that declares `depends_on` it, because the task graph, not the git graph, is the unit the system reasons about.

**Boundaries make autonomy safe.** Trust is earned through gates and then escalated, never assumed. Statuses default to `owner: user`; automation is something you declare, and the loader hard-fails on `owner: agent` with no agent named, because the whole point of the split is to make automation explicit. Zen is a per-session runtime toggle, not a config file, and it never silently escalates: an orchestrator crash auto-disables it for an hour, and projects with review workspaces never auto-merge review-routed tasks. The example that settles arguments: Zen's merge bar is a gated sequence (local checks green, no conflicts, diff under thresholds, no danger-path matches, CI green), and the first failed gate stops the merge and surfaces why. Autonomy is a privilege the gates keep re-earning.

**Prose where judgment lives, schema where machines act.** Configuration splits by who consumes it. Policy that requires judgment, like when to promote a backlog task or what clears the merge bar, lives in the orchestrator's instructions as editable prose. Behavior a machine executes, like transitions, actions, and status categories, lives in YAML with closed enums that the loader validates. The Zen design states it directly: the Rust side never inspects the merge conditions; the orchestrator prompt is where the user owns the policy. This settled where transition actions went (a closed enum in workflow YAML: `push_branch`, `open_pr`, `merge`, and so on) versus where the decision to fire them went (the orchestrator's category-keyed reasoning, in prose the user can edit per project).

**Built on boring, inspectable substrates.** tmux is the runtime, git worktrees are the isolation boundary, SSH is the transport, and `gh` is the merge machinery. We do not invent a primitive where a thirty-year-old one exists, because when something breaks the user can see it and fix it by hand with tools they already know. It settled remote worker communication: rather than a new network listener or an agent protocol, the hub reuses its existing SSH connection with a reverse socket forward, so remote workers get zero-install and the trust model is the SSH trust you already have. It also settled review workspaces: the dev server is a tmux `split-window` pane next to the agent, not a process manager.

**Fail loudly, never silently.** When automation hits ambiguity, it hard-fails with a message rather than guessing, and every decision the system makes lands in `events.log` where the user can audit it. Deprecations warn on stderr for one release and then a follow-up task removes the legacy path; we do not carry dual code paths forever. The settling example is dispatch: `sync_worktree()` aborts on a dirty worktree instead of stashing or discarding, and dispatch verifies the worktree is on the task's branch before any pane is touched, hard-failing if not. A wrong guess in an autonomous system compounds; a loud stop costs one round trip.

**Developer-flat, in copy and in behavior.** No grandeur, no multi-order benefit claims, and every claim states a mechanism; if a sentence has no feature behind it, cut it. This applies to the product itself, not just the homepage: vocabulary is functional (a workspace is a slot on a machine, not a "teammate"), and limits are acknowledged where they exist. It settled the positioning fact-check: the "adversarial review, QA, security" agents are built with shipped primitives, not shipped presets, so copy phrases them as what you can add, not what comes in the box. When honesty and polish conflict, honesty wins, because the champion is a developer and developers check.
